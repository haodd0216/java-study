## Positioning

这是「流程控制」的第一个纵深方向。基础篇学会了 if-else 的语法，而这篇回答：**写了很多 if-else 之后，如何让代码更容易读、更容易改、更不容易出错。**

## Analogy

多层 if-else = 老式电话总机接线，几十根线缠在一起。重构的目标：把混乱的总机接线换成清晰的电路板，每个条件独立走线，插拔不影响全局。

## Core Mechanism

### 1. 卫语句（Guard Clause）

核心理念：**先把「不合格的」踢出去，主逻辑放在方法末尾、最浅缩进层级。**

```java
// ❌ 箭头型
if (user != null) { if (user.isActive()) { ... } else { return 0; } } else { return 0; }

// ✅ 卫语句
if (user == null)       return 0;
if (!user.isActive())   return 0;
// 主逻辑
```

### 2. 三元运算符

语法：`条件 ? true值 : false值`

规则：
- 简单值二选一 → 用三元（`int max = a > b ? a : b`）
- 执行操作/嵌套/多行逻辑 → 用 if-else
- **永远不要嵌套三元**（可读性灾难）

### 3. 短路求值

`&&` 左边 false 则不执行右边；`||` 左边 true 则不执行右边。利用短路避免不必要的 null 检查嵌套：

```java
// ✅ 短路求值替代嵌套 if
if (user != null && user.isActive() && user.hasPermission("admin")) {
    doSomething();
}
```

### 4. if-else 链 → Map 查表

超过 3-4 个分支且都在比较同一个变量 → 用 Map 替代：

```java
// ✅ Map 替代 if-else 链
private static final Map<String, String> MAP = Map.of("#FF0000", "红色", ...);

public String get(String code) {
    return MAP.getOrDefault(code, "未知");
}
```

好处：新增改数据不改逻辑。

### 5. if vs switch

| 特征 | if-else | switch |
|------|---------|--------|
| 范围判断（>, <） | ✅ | ❌ |
| 离散值匹配 | 可以 | ✅ |
| 分支 < 3 | ✅ | 杀鸡用牛刀 |
| 分支 3-10 | 均可 | ✅ 更清晰 |
| 多值同逻辑 | `if(x==1\|\|x==2)` | `case 1,2 ->` |

## Code Example

会员折扣计算——四种写法对比：

```java
// ❌ 箭头型（嵌套地狱）
// ✅ 卫语句 + Map 查表 + 常量提取
private static final Map<String, double[]> RATE_MAP = Map.of(
    "VIP",  new double[]{0.1, 0.2},
    "SVIP", new double[]{0.2, 0.3}
);
private static final double THRESHOLD = 500.0;

public static double discount(String level, double amount) {
    if (level == null || amount <= 0) return 0;              // 卫
    double[] rates = RATE_MAP.get(level);
    if (rates == null) return 0;                              // 卫
    return amount * (amount > THRESHOLD ? rates[1] : rates[0]);  // 主逻辑
}
```

## Common Misconceptions

1. 「卫语句 = 多个 return 是坏习惯」→ C 时代旧观念，现代 Java 里卫语句让代码更清晰
2. 「三元嵌套省行数」→ 嵌套三元是可读性杀手第一名
3. 「Map 替代 if-else 是炫技」→ 3 个分支用 if-else 合理，30 个分支用 Map 是正确选择
4. 「switch 天然比 if-else 快」→ 性能差异可忽略，选语法的首要标准是语义清晰度

## Socratic Check

1. 7 层 if-else 嵌套，第一步重构？→ 先写卫语句，把失败/边界情况提前 return
2. `if (a != null && a.getValue() > 10)` a 为 null 会怎样？→ 不会抛异常，&& 短路不执行右半边
3. 什么时候不该用三元？→ 需要执行操作、嵌套、多行逻辑、需要提前 return 时
