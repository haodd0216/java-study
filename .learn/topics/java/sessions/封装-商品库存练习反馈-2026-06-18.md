1. 对 `stock` 和 `soldCount` 没有提供 setter，说明你理解了库存和销量应该由业务动作改变。

4. 你主动发现了 `getSalesAmount()` 的业务建模问题：中途改价后，历史销售额会按新价格重新计算。

## 关于 salesAmount 的关键问题

你发现的问题非常重要。

当前设计：

```java
public double getSalesAmount() {
    return this.price * this.soldCount;
}
```

这个设计表达的是：

> 当前单价 × 已售数量。

它适合“估算当前价格下这些销量值多少钱”，但不适合表示真实历史销售额。

如果商品先以 `19.9` 卖出 3 件，然后价格改成 `25.5`，当前方法会返回：

```text
25.5 * 3 = 76.5
```

但真实历史销售额应该是：

```text
19.9 * 3 = 59.7
```

所以这说明：

- 如果需求是“根据当前价格和销量计算一个展示值”，可以不定义 `salesAmount` 字段；
- 如果需求是“记录真实历史销售额”，就应该定义一个字段，例如 `private double salesAmount;`，并在 `sell` 方法里累加当时的销售额。

例如：

```java
private double salesAmount;

public void sell(int count) {
    if (count <= 0) {
        System.out.println("销售数量必须大于0");
        return;
    }
    if (count > stock) {
        System.out.println("库存不足");
        return;
    }
    stock -= count;
    soldCount += count;
    salesAmount += price * count;
}

public double getSalesAmount() {
    return salesAmount;
}
```

这就是封装里很关键的一点：

> 字段不是越少越好，而是要看这个值是不是对象必须保存的真实状态。

## 一个小细节

构造方法中当前写法是：

```java
this.name = name;
this.setPrice(price);
this.restock(stock);
this.soldCount = 0;
```

`name` 没有复用 `setName(name)`，所以如果构造时传入空名称，不会触发校验。

如果要完全复用规则，可以写成：

```java
setName(name);
setPrice(price);
restock(stock);
soldCount = 0;
```

不过 `restock(stock)` 会拒绝初始库存为 0，而题目要求是库存不能小于 0，所以如果允许初始库存为 0，构造方法里需要单独判断 `stock < 0`，不能直接复用 `restock`。

这次练习完成得很好，尤其是你主动发现“计算值”和“历史状态”的区别，说明你已经开始从代码语法进入对象建模思维。

封装这一节不仅是 private/getter/setter，更重要的是判断：

- 哪些值是字段；
- 哪些值是计算结果；
- 哪些变化必须通过业务方法发生；
- 哪些业务历史需要保存。
